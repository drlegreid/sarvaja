# P10.7: Entity Hierarchy Review - Typed References (Option B)

**Date:** 2024-12-27
**Status:** COMPLETED - OPTION B SELECTED
**Related:** GAP-ARCH-007, DECISION-003

---

## Summary

This review examines entity relationships in the TypeDB schema and implements **Option B: Typed References** where:
- Generic attributes (like `gap-reference`) become typed relations
- Decisions are task outcomes with authority attribute
- All cross-entity references use explicit typed relations

**Decision:** Implement Option B - Typed references for all entity links.

---

## Option B Design: Typed References

### Core Principle

Replace generic string attributes with typed relations that enable:
1. **TypeDB Inference**: Query traversals across entity types
2. **Data Integrity**: Referential constraints at schema level
3. **Semantic Clarity**: Relation types express purpose (e.g., `references-gap` vs generic string)

---

## Implementation Changes

### 1. Task-Gap Reference (Replace `gap-reference` attribute)

**Before (Generic Attribute):**
```typeql
task sub entity,
    owns gap-reference;  # String like "GAP-DATA-001"
```

**After (Typed Relation):**
```typeql
# Relation for task referencing a gap
references-gap sub relation,
    relates referencing-task,
    relates referenced-gap;

task sub entity,
    plays references-gap:referencing-task;

gap sub entity,
    plays references-gap:referenced-gap;
```

### 2. Decision as Task Outcome (Authority Link)

**Before (Separate Entities):**
```typeql
decision sub entity,
    owns decision-id, ...;

task sub entity,
    owns task-id, ...;
# No relationship between them
```

**After (Task Outcome Relation):**
```typeql
# Decision is outcome of a task with authority attribute
task-outcome sub relation,
    relates outcome-decision,
    relates source-task,
    owns has-authority;  # Boolean: true = authoritative decision

has-authority sub attribute, value boolean;

decision sub entity,
    plays task-outcome:outcome-decision;

task sub entity,
    plays task-outcome:source-task;
```

### 3. Complete Typed Reference Set

| Relation | From | To | Purpose |
|----------|------|----|---------|
| `references-gap` | task | gap | Task addresses a gap |
| `task-outcome` | decision | task | Decision resulted from task (with authority) |
| `implements-rule` | task | rule | Task implements a rule (existing) |
| `completed-in` | task | work-session | Task completed in session (existing) |
| `decision-affects` | decision | rule | Decision affects rule (existing) |
| `gap-rule-reference` | gap | rule | Gap references governing rule (existing) |

## Migration Steps

### Phase 1: Add New Relations (Non-Breaking)
1. Add `references-gap` relation to schema.tql
2. Add `task-outcome` relation to schema.tql
3. Add `has-authority` attribute to schema.tql
4. Add role players to task, decision, gap entities

### Phase 2: Data Migration
1. Query existing tasks with `gap-reference` attribute
2. For each task with gap-reference:
   - Match the referenced gap entity
   - Create `references-gap` relation
3. Query decisions linked to tasks (via session or manual mapping)
4. Create `task-outcome` relations with `has-authority: true`

### Phase 3: Deprecate Old Attribute
1. Stop writing to `gap-reference` attribute
2. Update client.py to use relation queries
3. Remove `gap-reference` from schema (TypeDB 3.x allows attribute removal)

## Query Examples

### Find Tasks Addressing a Gap
```typeql
match
    $t isa task, has task-id $tid;
    $g isa gap, has gap-id "GAP-DATA-001";
    (referencing-task: $t, referenced-gap: $g) isa references-gap;
get $tid;
```

### Find Authoritative Decisions from Tasks
```typeql
match
    $d isa decision, has decision-id $did;
    $t isa task, has task-id $tid;
    (outcome-decision: $d, source-task: $t) isa task-outcome, has has-authority true;
get $did, $tid;
```

### Infer Gap Resolution Path
```typeql
# Find task that resolved a gap via decision
match
    $g isa gap, has gap-id $gid;
    $t isa task;
    $d isa decision, has decision-id $did;
    (referencing-task: $t, referenced-gap: $g) isa references-gap;
    (outcome-decision: $d, source-task: $t) isa task-outcome, has has-authority true;
get $gid, $did;
```

## Benefits of Option B

1. **Full Traversability**: TypeDB can follow any reference path
2. **Inference Chains**: Decision→Task→Gap chains enable impact analysis
3. **Semantic Richness**: Authority attribute distinguishes decision types
4. **Schema Validation**: Invalid references caught at insert time
5. **Graph Queries**: Mermaid diagrams from relation traversal
